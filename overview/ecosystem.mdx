## The two-pole stack

Almost everything resolves to one of two technology poles. Knowing which pole a project sits on tells you most of what you need about its toolchain and conventions.

| Pole | Used for | Representative projects |
|---|---|---|
| **Next.js** (App Router, canary) | All web and landing surfaces | `greekcore/web`, `greekcore/landing`, Relic landing, Mavri landing, `personal-site` |
| **Python** (`pyproject`, `uv`) | AI backends and data pipelines | `relic-core`, `mavri/studio`, `agent-simulator` backend |

<Note>
  The Next.js repos run bleeding-edge **canary** builds. Each carries a standard `nextjs-agent-rules` block reminding agents: *"This is NOT the Next.js you know — read `node_modules/next/dist/docs/` first."*
</Note>

## How a venture is structured

The most sophisticated ventures share a common shape, modeled most fully by GreekCore v2:

<Steps>
  <Step title="A workspace orchestrator">
    A parent repo owns only cross-cutting tooling. A single front door (`mise.toml`) pins the toolchain and defines every dev task.
  </Step>
  <Step title="Independent child repos">
    A `repos.manifest` clones the real repos (`api`, `web`, `infra`, `docs`) — gitignored, not submodules. The workspace stays thin.
  </Step>
  <Step title="One-command local stack">
    `mise run setup` clones, installs, and fans out secrets; `mise run dev` boots Postgres, migrations, seeds, and every app process under `process-compose`.
  </Step>
  <Step title="Agent-agnostic rules">
    A one-line `CLAUDE.md` imports a shared `AGENTS.md` — a single source of truth carried identically across every repo in the venture.
  </Step>
</Steps>
