**Pelagic AI** (the `agent-simulator` codebase) is a satellite-imagery-driven environment simulator that orchestrates LLM agents via the Model Context Protocol (MCP) to generate realistic airport operational scenarios — using real-world OpenStreetMap (OSM) data and satellite-quality predictions.

## Tech stack

- **Backend:** FastAPI (async Python 3.11), SQLite (aiosqlite + SQLAlchemy async)
- **Frontend:** Next.js 15 (React 18, TypeScript 5, App Router)
- **Graph & pathfinding:** NetworkX
- **Agents:** Model Context Protocol via FastMCP; LangGraph + LangChain
- **Visualization:** Mapbox GL JS + Three.js (custom GLSL shaders)
- **Orbital & solar math:** `pvlib`, `skyfield`, `sgp4`

## Engineering deep dives

### 1. Dual-loop agent workflows via LangGraph

Automates layout discovery and flight execution by looping until complete, letting specialized models share conversation state.

- The **initialization graph** fetches runways, taxiways, and gate properties through MCP tools, looping until no further tools are called.
- The **simulation timestep graph** commands plane operations on scheduling rules, queries final positions, and feeds environmental statistics to a high-reasoning model (e.g. `o3-mini`) that generates visual satellite observation reports.

<Note>
  **Reference:** `backend/llm_pipeline/surveillance.py` — the `AirportSimulationService` orchestrator binding layout and movement tools to their specialized agents.
</Note>

### 2. Custom WebGL 3D aircraft layer with Fresnel shaders

Renders 3D aircraft (Boeing 737, Airbus A330/A380, F18, F22, Tie Fighter, X-Wing) directly on Mapbox GL JS with a night-preset style and a glowing red effect.

It subclasses `mapboxgl.CustomLayerInterface` inside a React client component, maps coordinates into Mercator space, manages a `THREE.WebGLRenderer` over the Mapbox canvas, and compiles custom shaders implementing a fresnel glow factor:

$$\text{fresnel} = (1.0 - |\vec{N} \cdot \vec{V}|)^2$$

<Note>
  **Reference:** `frontend/src/components/simulation/MapVisualization.tsx` — model loading, GLSL shader compilation, and entity coordinate calculations.
</Note>

### 3. Scenario-isolated SQLite, three tables per scenario

Provides strict isolation between simulation scenarios while staying on a lightweight, file-based SQLite database.

A dynamic database manager creates per-scenario tables at runtime — `scenario_{uuid}_osm_features`, `scenario_{uuid}_entities`, and `scenario_{uuid}_schedule` — alongside a global `scenario_metadata` table.

<Note>
  **Reference:** `backend/database/manager_v2.py` — dynamically creates and drops scenario-specific SQLite tables via SQLAlchemy factories.
</Note>

### 4. Satellite environmental quality evaluator

Determines whether a satellite pass over an airport would yield clear imagery, based on weather, solar coordinates, and cloud cover.

It models satellite passes from TLE data with `skyfield` and `sgp4`, evaluates the solar elevation angle (computing a 1 m pole's shadow length), integrates cloud fractions, and combines them into a weighted score:

| Factor | Weight |
|---|---|
| Solar | 35% |
| Cloud | 40% |
| Weather | 25% |

<Note>
  **Reference:** `backend/imaging_assessment/env_assessment.py` — scoring logic and optimal-window identification.
</Note>

### 5. Node snapping & Haversine collision avoidance

Snaps moving planes onto taxiway networks and prevents physical overlaps.

The `move_aircraft` MCP tool queries the airport's NetworkX graph: within 10 m of a graph node, it snaps the aircraft to the node; otherwise it leaves it on a free coordinate and computes bearing from the last point. It computes the Haversine distance to other active aircraft and **rejects moves below a 35 m minimum safe threshold**.

<Note>
  **Reference:** `backend/mcp_toolkit/tools/airport_controller_tools.py` — snapping, bearing, and collision warnings.
</Note>

## The hardest problem

**Blending the Mapbox GL JS render loop with the Three.js WebGL context inside React.** Without careful cleanup, canvas initialization leaks resources or drops frames. The fix: pre-calculate translation matrices relative to a fixed airport-origin node and update active sprite coordinates synchronously during the `render` pass of Mapbox's frame ticks.
